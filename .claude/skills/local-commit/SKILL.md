# local-commit - 智能本地提交

这个技能帮助你智能地提交代码变更到本地 Git 仓库（不推送到远程）。
自动分析代码变更，生成简洁的中文提交信息，并提交到本地仓库。支持识别修改类型（feat/fix/refactor 等）和相关模块。

## 使用方式

```bash
/local-commit          # 不更新版本号，直接提交
/local-commit -b       # 更新主版本号 (MAJOR): 1.0.0 -> 2.0.0
/local-commit -s       # 更新次版本号 (MINOR): 1.0.0 -> 1.1.0
/local-commit -p       # 更新修订号 (PATCH): 1.0.0 -> 1.0.1
```

## 何时使用

- 快速保存本地开发进度
- 批量完成多个任务后再统一推送
- 离线开发或网络不稳定时
- 需要频繁提交但不希望每次都推送
- 需要在提交时同步更新应用版本号

## 功能

1. 根据参数可选地更新版本号（-b/-s/-p）
2. 分析所有修改的文件
3. 查看代码差异内容
4. 根据修改内容智能生成简洁的中文提交信息
5. 添加文件到暂存区
6. 创建提交到本地
7. **不执行 git push**

## 执行步骤

### 步骤 0：检查并更新版本号（如果指定了参数）

检查 `args` 参数，判断是否需要更新版本号：

- **`-b` 参数**：更新主版本号 (MAJOR)，如 `1.0.0` -> `2.0.0`
- **`-s` 参数**：更新次版本号 (MINOR)，如 `1.0.0` -> `1.1.0`
- **`-p` 参数**：更新修订号 (PATCH)，如 `1.0.0` -> `1.0.1`
- **无参数**：不更新版本号，直接跳到步骤 1

如果指定了版本号更新参数，执行以下操作：

1. 读取 `pyproject.toml` 文件，找到当前版本号（在 `[project]` 部分的 `version` 字段中）
2. 根据参数类型更新对应的版本号部分：
   - 解析当前版本号（格式：`major.minor.patch`）
   - 根据参数递增对应部分
   - 将更新后的版本号写回 `pyproject.toml`
3. 验证版本号格式正确（使用正则表达式 `\d+\.\d+\.\d+`）
4. 将版本号文件的修改也纳入提交范围

### 步骤 1：查看修改状态

首先运行 `git status` 查看所有修改的文件：
```bash
git status
```

### 步骤 2：查看代码差异

运行 `git diff` 查看具体的代码修改内容：
```bash
git diff
```

如果需要查看已暂存的修改：
```bash
git diff --staged
```

### 步骤 3：分析修改内容

根据以下规则分析修改类型：

**修改类型判断：**
- **feat (✨)**: 新增功能，关键词：`添加`, `新增`, `feat`, `feature`
- **fix (🐛)**: 修复问题，关键词：`修复`, `fix`, `bug`, `错误`
- **refactor (♻️)**: 重构代码，关键词：`重构`, `refactor`, `优化`
- **docs (📝)**: 文档修改，关键词：`文档`, `doc`, `README`
- **test (✅)**: 测试相关，关键词：`测试`, `test`
- **chore (🔧)**: 配置/构建，关键词：`配置`, `依赖`, `chore`

**涉及的模块识别：**
- 从文件路径提取主要模块名称（如 `gateway/`, `tools/`, `policy/` 等）
- 如果涉及多个模块，取前 2 个

### 步骤 4：生成提交信息

根据分析结果生成提交信息，格式：

```
<emoji> <type>(<模块>): <简短描述>

Co-Authored-By: Codex <noreply@openai.com>
```

示例：
- `✨ feat(gateway): 新增 WebSocket 事件流`
- `🐛 fix(tools): 修复文件读取权限检查`
- `♻️ refactor(policy): 优化策略引擎性能`
- `📝 docs: 更新 CLAUDE.md 文档`

### 步骤 5：执行提交

```bash
# 添加所有修改
git add .

# 创建提交（使用生成的提交信息）
git commit -m "<提交信息>"
```

**注意：此技能不执行 git push**

## 示例

### 示例 1：普通提交（不更新版本号）

假设修改了 `src/jaxbot/gateway/ws.py` 和 `src/jaxbot/tools/executor.py`，添加了新的事件处理：

```bash
/local-commit
```

1. 检查参数：无参数，跳过版本号更新
2. 查看 git status 确认修改
3. 查看 git diff 了解具体改动
4. 分析：新增功能 → feat，涉及 gateway 和 tools 模块
5. 生成提交信息：`✨ feat(gateway, tools): 新增事件处理逻辑`
6. 执行 git add 和 git commit

### 示例 2：提交并更新次版本号

完成了一个新功能，需要发布小版本更新：

```bash
/local-commit -s
```

1. 检查参数：`-s` 表示更新 MINOR 版本号
2. 读取 `pyproject.toml`，当前版本为 `0.0.1`
3. 更新版本号：`0.0.1` -> `0.1.0`
4. 将版本号文件修改纳入提交范围
5. 正常执行提交流程

### 示例 3：提交并更新修订号

修复了一个 bug，需要发布补丁版本：

```bash
/local-commit -p
```

1. 检查参数：`-p` 表示更新 PATCH 版本号
2. 读取当前版本 `0.1.0`
3. 更新版本号：`0.1.0` -> `0.1.1`
4. 验证并提交

## 后续推送

当需要推送到远程仓库时，可以手动执行：
```bash
git push
```

## 注意事项

- 如果没有修改，提示用户"没有需要提交的代码"
- 提交信息应该简洁明了，突出主要改动
- 始终包含 Co-Authored-By 标记
- 此技能只提交到本地，不会推送到远程仓库
- 版本号参数说明：
  - `-b` (Major)：重大变更，不兼容的 API 修改
  - `-s` (Minor)：向下兼容的功能性新增
  - `-p` (Patch)：向下兼容的问题修正
- 版本号更新会自动修改 `pyproject.toml` 文件并纳入提交
- 如果版本号格式不正确或文件不存在，提示用户错误并中止提交
