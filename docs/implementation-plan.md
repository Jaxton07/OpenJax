# openJax 实施计划（第一版）

## 1. 目标与范围

### 1.1 总目标
在全新仓库 `openJax` 中，先构建一个可在终端交互运行的最小 Agent，保留以下基础能力：

- 回合式 agent loop（用户输入 -> 模型输出 -> tool 调用 -> 回写上下文）
- 本地文件读取能力（read/list/grep）
- 本地 shell 命令执行
- 审批机制（approval）
- 沙箱机制（sandbox）
- 基础补丁写入（apply_patch 等价能力）

### 1.2 当前阶段不做

- 系统级自动化（macOS 截屏、自动控制 UI）
- 长期记忆完整体系
- 大规模多 agent 协作
- 复杂 GUI/TUI

### 1.3 第一阶段验收标准

满足以下条件即可判定第一阶段完成：

1. 可在命令行持续对话并处理多轮任务。
2. Agent 能读目录、读文件、grep 文本。
3. Agent 能执行 shell 命令，且受审批与沙箱策略控制。
4. Agent 能进行受控文件修改（补丁方式）。
5. 关键流程有最小集成测试（至少 3 个端到端场景）。

---

## 2. 路线选择（建议）

你已经决定新建仓库重做。建议采用“先可运行，再去耦”的路线：

1. **阶段 A：Bootstrap**
   先在 `openJax` 跑通最小骨架，哪怕暂时通过依赖或参考实现复用现有逻辑。
2. **阶段 B：稳定核心循环**
   固化协议、事件循环、工具注册与执行路径。
3. **阶段 C：渐进定制**
   在不破坏核心稳定性的前提下扩展新工具与平台能力。

核心原则：

- 不要一开始就全面复制全部代码。
- 每次只改一层（协议层/执行层/工具层）。
- 每个里程碑都保持“可运行、可测试、可回滚”。

## 3. 里程碑计划

## M0：开发环境就绪（你当前状态）

### 目标
让 macOS 上具备 Rust 工具链和基础构建能力。

### 任务

1. 安装 Xcode Command Line Tools（如未安装）。
2. 安装 Rustup + Cargo + Rust stable。
3. 安装基础开发工具：`git`、`just`（可选）、`rg`。
4. 验证命令：
   - `cargo --version`
   - `rustc --version`
   - `cargo new smoke_test && cargo run`

### 产出

- 本机可执行 `cargo build/run/test`。
- `openJax` 仓库可开始 Rust 开发。

---

## M1：最小 CLI + 回合循环（无高级功能）

### 目标
在 `openJax` 中建立最小可运行 CLI，先打通主流程。

### 任务

1. 创建项目结构（建议 workspace）：
   - `openjax-core`
   - `openjax-cli`
   - `openjax-protocol`
2. 在 `openjax-protocol` 定义最小 `Op/Event`：
   - `UserTurn`
   - `ToolCall`
   - `ToolResult`
   - `TurnCompleted`
3. 在 `openjax-core` 实现 submission loop：
   - 接收 `Op`
   - 调用模型
   - 解析 tool call
   - 执行 tool
   - 继续循环直到回合结束
4. 在 `openjax-cli` 实现 REPL：
   - 用户输入
   - 打印事件流
   - 显示最终答复

### 验收

- 能在终端输入一句话并得到响应。
- 若模型返回 tool call，系统可继续流转而不崩溃。

---

## M2：基础工具能力（read/list/grep + shell）

### 目标
接入最核心本地操作能力。

### 任务

1. 工具注册机制：
   - `ToolSpec`
   - `ToolRouter`
   - `ToolHandler`
2. 实现三个只读工具：
   - `read_file`
   - `list_dir`
   - `grep_files`
3. 实现 `exec_command`：
   - 同步/异步执行
   - stdout/stderr 捕获
   - 超时控制
   - 输出截断
4. 加入 tool 调用审计日志（最小版）：
   - 时间
   - 命令/路径
   - 退出码

### 验收

- Agent 可自主读文件、列目录、搜索文本。
- Agent 可执行 shell 命令并返回结构化结果。

## M3：审批与沙箱（第一版）

### 目标
在“能跑”的基础上引入安全控制，不做过度复杂化。

### 任务

1. 审批策略（至少三档）：
   - `always_ask`
   - `on_request`
   - `never`
2. 执行前审批流程：
   - 高风险命令提示
   - 用户确认/拒绝
3. 沙箱模式（最小两档）：
   - `workspace_write`
   - `danger_full_access`（仅开发用）
4. 基础网络策略：
   - 默认禁网或白名单
5. 文件写入策略：
   - 仅允许工作目录
   - 路径规范化与逃逸检查（`..`、符号链接）

### 验收

- 被禁止命令不会执行。
- 越权路径写入被阻止。
- 审批提示清晰且可复现。

## 4. 你后续能力扩展路线（M6+）

> 以下为后续规划，当前不进入实现阶段。

### 4.1 macOS 系统能力（截图/自动化）

建议以独立工具进程形式接入（MCP 或本地 RPC），不要直接塞进 core：

- `mac_screenshot.capture`
- `mac_file.reveal`
- `mac_automation.run`（需严格权限控制）

并配套：

- 单独审批等级
- 调用审计日志
- 速率限制与超时

### 4.2 长期记忆

建议分三层：

1. `memory_write`：结构化写入（事实、偏好、项目规则）
2. `memory_retrieve`：按语义/标签召回
3. `memory_injection_policy`：在何时、注入多少、冲突如何处理

### 4.3 个性化工具生态

- 把你自定义能力都走统一 Tool API。
- 每个工具必须声明：
  - 风险级别
  - 所需权限
  - 是否可并发
  - 超时上限

### 4.4 多代理支持（已预留接口）

协议层和核心层已预留扩展接口：

- **协议层**：已实现 `ThreadId`, `AgentSource`, `MAX_AGENT_DEPTH`, `Op::SpawnAgent` 等
- **核心层**：已预留 `Agent::spawn_sub_agent()` 接口
- **后续实现**：需要实现多代理调度器和状态管理

### 4.5 流式通信（SSE/WebSocket）

- **当前**：HTTP 轮询（MVP 够用）
- **TUI 阶段**：升级为 SSE 流式（用户体验优先）
- **未来**：需要双向交互时升级为 WebSocket（如中断正在运行的 agent）

## 5. 风险与控制

### 主要风险

1. 一开始范围过大，导致长期无法“跑通”。
2. 安全策略太晚接入，后续返工巨大。
3. 工具协议频繁改动，造成上下游不稳定。

### 控制措施

1. 每个里程碑必须有演示脚本和验收用例。
2. 保持单一主分支可运行。
3. 每次只改一层，先写测试再扩展功能。
