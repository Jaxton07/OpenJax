# 08 生产灰度执行清单

## 使用范围
- 适用于 Streaming v2 + tool batch + gateway 恢复 + web 批处理渲染的生产灰度上线。
- 配套开关：
  - `OPENJAX_STREAM_ENGINE_V2`
  - `OPENJAX_TOOL_BATCH_V2`
  - `OPENJAX_SSE_PROTOCOL_V2`
  - `OPENJAX_WEB_STREAM_RENDER_V2`

## 角色分工
- 发布负责人：执行变更、记录阶段结果、触发回滚。
- 值班观察人：持续观察指标与告警，给出放量/停止建议。
- 业务验收人：抽检关键会话体验（流畅度、完整性、恢复能力）。

## 上线前检查（T-1）
- 代码版本固定，构建产物与镜像已签名。
- 以下验证在预发完成并留档：
  - `cargo test -p openjax-core`
  - `cargo test -p openjax-gateway`
  - `pnpm -C ui/web test`
  - `pnpm -C ui/web build`
- 验证开关读取生效：
  - core 识别 `OPENJAX_STREAM_ENGINE_V2`、`OPENJAX_TOOL_BATCH_V2`
  - gateway 识别 `OPENJAX_SSE_PROTOCOL_V2`
  - web 识别 `OPENJAX_WEB_STREAM_RENDER_V2`

## 阶段执行步骤

### 阶段 0：上线基线（0%）
- 操作：
  - 部署新版本到生产，但保持灰度入口不放量。
  - 开关按默认开启，路由权重保持 0%。
- 观察：
  - 服务启动日志是否包含开关状态与模型/路由信息。
  - 无新增错误峰值。

### 阶段 1：小流量（5%）
- 操作：
  - 将 v2 路径流量权重切到 5%。
  - 观测窗口至少 30 分钟。
- 必看指标：
  - TTFT 劣化 ≤ 15%
  - Delta p95 劣化 ≤ 20%
  - 缺口恢复失败率 < 0.5%
  - tool batch 失败率 < 2%
  - 前端 `response_error` 比例 < 1%
- 放量条件：
  - 全部指标达标且无 P1/P0 告警。

### 阶段 2：中流量（20%）
- 操作：
  - 权重从 5% 提升到 20%。
  - 观测窗口至少 60 分钟，覆盖一段业务高峰。
- 额外抽检：
  - 至少 10 个真实会话：含多工具批次、审批场景、断线恢复场景。
- 放量条件：
  - 指标持续稳定，且抽检无阻塞问题。

### 阶段 3：扩大流量（50%）
- 操作：
  - 权重提升到 50%，持续观测 1~2 个高峰窗口。
- 必看事件：
  - core：`final_stream_ttft`、`tool_batch_completed`
  - gateway：`sse_lagged_recovered`、`sse_lagged_recovery_failed`
  - web：`web_batch_flush`
- 放量条件：
  - 指标趋势稳定，无持续性上升风险。

### 阶段 4：全量（100%）
- 操作：
  - 权重切到 100%，保留旧路径开关 24~48 小时作为快速回退。
- 收尾：
  - 产出灰度总结：各阶段指标、问题、处理与结论。

## 回滚操作（任一阶段可执行）
- 触发条件（任一满足即回滚）：
  - TTFT 劣化 > 25% 且持续 10 分钟
  - 缺口恢复失败率 ≥ 1%
  - 前端 `response_error` ≥ 3%
  - 出现 P1/P0 业务故障
- 回滚顺序（由局部到全局）：
  1. `OPENJAX_WEB_STREAM_RENDER_V2=0`
  2. `OPENJAX_SSE_PROTOCOL_V2=0`
  3. `OPENJAX_TOOL_BATCH_V2=0`
  4. `OPENJAX_STREAM_ENGINE_V2=0`
- 回滚后动作：
  - 记录回滚时间、触发指标、影响范围。
  - 保留现场日志与样本请求，进入故障复盘。
