# 05 迁移与灰度策略

## 代码落地策略
- 采用原路径就地重构，不创建平行 v2 产品目录。
- 仅在必要边界新增局部模块文件，用于隔离新状态机与适配层。
- 通过运行时开关控制新旧路径切换。

## 阶段化迁移
1. 契约阶段：先完成协议与数据模型升级，保留旧路径兼容。
2. 引擎阶段：引入新回合引擎并在受控范围开启。
3. 传输阶段：gateway 开启 v2 恢复机制并观察缺口率。
4. 体验阶段：web 切到批处理渲染与新状态机。
5. 收敛阶段：默认新路径，旧路径降级为应急兜底。

## 灰度开关建议
- `OPENJAX_STREAM_ENGINE_V2`：开启新执行引擎。
- `OPENJAX_TOOL_BATCH_V2`：开启批量工具调度。
- `OPENJAX_SSE_PROTOCOL_V2`：开启 v2 事件协议。
- `OPENJAX_WEB_STREAM_RENDER_V2`：开启前端批处理渲染。

## 开关默认值与降级
- 默认值：上述四个开关默认开启（值为空时按开启处理）。
- 关闭语义：值为 `0/off/false/disabled` 时关闭。
- 降级顺序：`WEB_STREAM_RENDER_V2 -> SSE_PROTOCOL_V2 -> TOOL_BATCH_V2 -> STREAM_ENGINE_V2`。
- 回退动作：关闭对应开关后保持协议兼容，不中断会话与历史事件查询。

## 观测指标
- TTFT（首 token 延迟）
- Delta 间隔分布（p50/p95）
- 事件缺口率与恢复成功率
- 批量工具调用耗时与失败率

## 灰度阈值
- TTFT：相较基线劣化不超过 15%。
- Delta p95：相较基线劣化不超过 20%。
- 缺口恢复失败率：低于 0.5%。
- 批量工具失败率：低于 2%。
- 前端流式错误率（response_error）：低于 1%。

## 灰度步骤
1. 开发环境：四开关全开，执行 core/gateway/web 全量测试。
2. 预发环境：开启 `STREAM_ENGINE_V2 + TOOL_BATCH_V2`，观察 TTFT 与 batch 失败率。
3. 预发环境：再开启 `SSE_PROTOCOL_V2`，观察缺口恢复指标与终态一致性。
4. 预发环境：最后开启 `WEB_STREAM_RENDER_V2`，观察前端错误率与交互流畅度。
5. 生产环境：按 5% -> 20% -> 50% -> 100% 扩量，每步至少观察一个完整高峰窗口。

## 回滚策略
- 任一阶段异常升高时，按开关逐级回退。
- 保持协议与事件兼容，不做破坏性回滚。
