# 05 迁移与灰度策略

## 代码落地策略
- 采用原路径就地重构，不创建平行 v2 产品目录。
- 仅在必要边界新增局部模块文件，用于隔离新状态机与适配层。
- v2 主路径固定启用，旧路径不再作为运行时切换选项。

## 阶段化迁移
1. 契约阶段：先完成协议与数据模型升级，保留旧路径兼容。
2. 引擎阶段：引入新回合引擎并在受控范围开启。
3. 传输阶段：gateway 开启 v2 恢复机制并观察缺口率。
4. 体验阶段：web 切到批处理渲染与新状态机。
5. 收敛阶段：默认新路径，旧路径降级为应急兜底。

## 灰度开关建议
- `OPENJAX_WEB_STREAM_RENDER_V2`：开启前端批处理渲染。

## 开关默认值与降级
- 默认值：stream engine v2、tool batch v2、sse protocol v2 固定开启；web render v2 默认开启。
- 关闭语义：值为 `0/off/false/disabled` 时关闭。
- 降级顺序：`WEB_STREAM_RENDER_V2 -> 版本回退`。
- 回退动作：优先使用版本回退或分支回退，保持协议兼容，不中断会话与历史事件查询。

## 观测指标
- TTFT（首 token 延迟）
- Delta 间隔分布（p50/p95）
- 事件缺口率与恢复成功率
- 批量工具调用耗时与失败率

## 灰度阈值
- TTFT：相较基线劣化不超过 15%。
- Delta p95：相较基线劣化不超过 20%。
- 缺口恢复失败率：低于 0.5%。
- 批量工具失败率：低于 2%。
- 前端流式错误率（response_error）：低于 1%。

## 灰度步骤
1. 开发环境：默认配置执行 core/gateway/web 全量测试。
2. 预发环境：验证 v2 主路径稳定（stream/tool batch/sse 全启用默认），观察 TTFT 与恢复指标。
3. 预发环境：按需关闭 `WEB_STREAM_RENDER_V2` 做前端局部降级演练。
4. 预发环境：恢复默认，观察前端错误率与交互流畅度。
5. 生产环境：按 5% -> 20% -> 50% -> 100% 扩量，每步至少观察一个完整高峰窗口。

## 回滚策略
- 任一阶段异常升高时，优先关闭 `WEB_STREAM_RENDER_V2`，其后执行版本/分支回退。
- 保持协议与事件兼容，不做破坏性回滚。
