# 01. 目标与决策

## 1. 背景与现状问题

基于近期 WebUI 实测：

1. 当前链路在“planner JSON 流 + 二次封装”路径下，流式输出存在体感卡顿。
2. 切换到更直接的 provider 流后，流畅度明显提升，说明瓶颈主要在后端热路径处理，而非前端渲染本身。
3. 现有流程在“纯文本回复”和“工具调用编排”之间边界不够清晰，导致所有请求被重型路径拖慢。

## 2. 设计目标

1. 性能目标
   - 降低 TTFT（首字节到首 token 时间）。
   - 降低输出抖动（减少中间重组和冗余事件）。
   - 保证输入区与非流式区域稳定。
2. 架构目标
   - 纯文本场景默认高性能直通。
   - 工具调用场景保持完整编排与审批能力。
   - 数据面与控制面解耦。
3. 工程目标
   - 可灰度启用，可快速回退。
   - 与现有 gateway/WebUI 协议兼容演进。

## 3. 非目标

1. 本阶段不重写工具系统与审批系统语义。
2. 本阶段不引入复杂多会话 UI/多租户能力。
3. 本阶段不做 Markdown 富渲染优化，先聚焦纯文本流体验与架构稳定。

## 4. 最终设计决策

1. 采用“单入口分发器 + 双分支执行器”。
2. 主流程是串行路由，不是并行混流。
3. 分发器只负责“判定 + 路由 + 锁定分支”，不负责审批。
4. 默认走 `Fast Path`，仅在确认工具调用意图时进入 `Orchestrator Path`。
5. 判定依据以“结构化 tool-call 信号”为主，不依赖自然语言猜测。
6. 为降低误判与延迟，允许极短判定缓冲窗（如 50~120ms）。

## 5. 为什么不是“全并行边流边判”

1. SSE 文本一旦下发前端不可回滚。
2. 若误把工具调用片段当文本发送，会污染对话内容并破坏用户心智。
3. 并行混流对一致性与排障成本过高，不适合作为默认路径。

结论：使用“短缓冲 + 早判定 + 分支锁定”比“并行混流”更稳妥。
