# 03 非功能需求（NFR）与约束

## NFR 定义

NFR（Non-Functional Requirements）用于定义系统质量目标，不描述业务功能本身。

## v1 NFR 目标（用于 phase-5 验收）

### 性能

- `POST /sessions/{id}/turns` 网关处理延迟：P95 <= 1500ms（不含超长模型生成时长）。
- SSE 首包时间（建立连接到首个事件）：P95 <= 1000ms。
- 单实例并发活跃 SSE 连接：>= 200（个人部署目标）。

### 稳定性

- 非预期 5xx 错误率：< 1%（按小时窗口）。
- SSE 断连后可恢复时间：<= 5s（在服务可用前提下）。
- 同一会话事件 `event_seq` 保序一致性：100%。

### 可维护性

- 代码与文档边界一致：协议只在 phase-2 定义。
- 变更必须可追溯：关键变更写入 `DECISIONS.md`。
- 文档读取遵循低上下文规则，默认不跨阶段。

### 安全基线

- 所有业务 API 默认鉴权（API Key）。
- 日志禁止输出 API Key 明文。
- 错误响应不暴露内部路径与堆栈。

## 约束

- 不重写 Rust 内核。
- v1 不支持多租户（仅单租户个人部署）。
- v1 不引入 WebSocket/JWT/daemon 双轨兼容。
- 当前阶段输出设计文档，不实施代码改动。

## 验收映射

- phase-2：冻结可测协议与错误语义。
- phase-3：落实状态模型、故障策略、观测字段。
- phase-5：基于本 NFR 执行验收与发布门禁。
